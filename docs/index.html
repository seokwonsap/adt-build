01The big idea

SAP's Eclipse ADT talks to the system over a documented HTTP API (/sap/bc/adt/…). Anything Eclipse does — create, lock, write source, activate, run unit tests, ATC — is an HTTP call. So an AI agent can drive the whole authoring loop directly, with a thin CLI in between and the live system as the compiler.

03The verify loop

Activation returns a verdict — but false has two meanings. Reading the message is the whole game.

flowchart TD A["POST /activation"] --> Q{"activationExecuted?"} Q -->|true| G["✅ compiled — run ATC + unit"] Q -->|"false + type=E message"| E["❌ real syntax/type error
→ read msg + href#start=line,col → fix → rebuild"] Q -->|"false + NO message"| N["🟢 nothing to activate
= source identical to active version
(benign no-op, common on rebuild)"] E --> A G --> D{"ATC clean?
unit green?"} D -->|no| E2["fix findings"] --> A D -->|yes| DONE(["🏁 done — Clean Core grade A–D"])
Exit-code gate: 0=PASS · 1=compile/activate · 2=unit · 3=ATC — so an agent can branch on it.

The empty-result trap

Symptom ABAP Unit / activation returns a schema-valid empty result with HTTP 200 → reads as green.
Cause An empty <runResult/> means nothing ran, not "passed"; a namespaced <options> is silently dropped.
Fix No <program> element = "no tests found", never "passed". Keep <options> unqualified.

Read the message before declaring a boundary

Lesson Most "needs Eclipse" walls were self-inflicted bugs.
Cause A generationExecuted="false" often carries a message that names the real fix.
Fix Always parse the activation reply's messages; for view entities that flag isn't even a failure — verify with an actual query.

04Two ways in — and when to use which

An MCP server (VSP) is great for read/search/test; raw ADT REST is the trustworthy build path. The catalog was written running both.

read · search · test

VSP MCP

One SAP(action,…) tool for read, search, ABAP Unit, ATC, system info, and (partial) debugger. Convenient in-agent. But create can time out on a busy package, and it can't make DOMA/DTEL at all.

create · activate · run

raw ADT REST — abap CLI

The reliable path for create → activate → run. No MCP, no Eclipse — just HTTP. Works from anywhere the system is reachable, scriptable, auditable, and fills VSP's DDIC gaps.

Rule of thumb

Build (create+activate+run) via raw REST; read/edit/test via VSP or raw REST. When VSP can't do something, pull the exact create media type from the discovery <app:accept> of its collection — they're the unversioned application/vnd.sap.adt.<x>Source+xml form.

06The gotchas that cost the most time

Ten highlights. The full symptom → cause → fix catalog (A–P, dozens of entries) lives in LEARNINGS.md.

G1 · Wrong activation endpoint

Symptom Raw-REST activation returns 304, activates nothing.
Cause /activation/inactiveobjects is for listing.
Fix POST the bare /sap/bc/adt/activation?method=activate.

G2 · CSRF token rotates

Symptom Activate 403s right after LOCK/PUT.
Cause A modifying op in a stateful session rotates the token.
Fix Re-fetch the token (fresh session) before activate; safest before every create/lock/PUT.

G3 · Circular CDS won't activate

Symptom Root ↔ child composition (or DDLS ↔ AMDP class) zombies — active but no runtime.
Cause One-at-a-time activation never resolves the cycle.
Fix Mass-activate both in a single /activation call.

G4 · Dynamic SELECT FROM (var)

Symptom Inline @DATA() fails activation with a dynamic FROM.
Fix CREATE DATA r TYPE TABLE OF (name) + field-symbol; inline @DATA only with a static FROM.

G5 · Client-dependent AMDP TF

Symptom Activation cancels: "type field at pos 1 is CHAR (not CLNT)".
Fix Put a abap.clnt key field first in the return structure.

G6 · Function-module source shape

Symptom PUTting the classic *" interface block → 400.
Fix Use an inline signature (like a method): FUNCTION z_x IMPORTING VALUE(iv) TYPE …. Activate the FM ref (FUGR/FF), not the group.

G7 · DOMA / DTEL have no source/main

Symptom PUT to source/main → 404.
Fix PUT the full object XML to the object URI; element presence + order is strict — model it on a live GET of a real object.

G8 · V4 binding needs publishing

Symptom OData 404 "service group not published" after activating SRVB.
Fix POST /businessservices/odatav4/publishjobs with an objectReferences body. Also: binding_version:"V4" + category:"0" (=UI).

G9 · ATC default = cloud-readiness

Symptom On-prem APIs (number ranges, DATASET IO, sy-datum) flag as P1/P2.
Cause The system's default variant is a restricted-scope (cloud) one.
Fix Objects still activate. Treat as advisory (--atc-max-prio 0) or move to released alternatives behind a seam.

G10 · Green ≠ correct

Symptom Code activates + passes ATC but returns wrong values (e.g. integer division always 0 → a false RED).
Cause Activation proves compile-correctness, not logic.
Fix Adversarial review + a boundary test harness catch logic-correct-but-wrong-value bugs. Do a second pass on anything with threshold/state logic.

07The headless boundary verdict

🏁 Every "needs Eclipse" wall fell.

AMDP table functions, composition CDS, and DCL access controls were each once believed to require Eclipse. All three build, activate, and run fully headless via raw ADT REST — each "boundary" turned out to be a bug (wrong endpoint, one-at-a-time vs mass activation, or wrong create media type), not a platform limit.

✓ managed / related-root RAP BOs ✓ CDS views · joins · hierarchies ✓ AMDP table functions ✓ composition CDS ✓ DCL row-level security ✓ OData V4 (publish) ✓ classrun · RTTI · parallel ✓ classic reports · FM · OO · GUI-ALV ✓ message classes ✓ DDIC: TABL · STRU · DOMA · DTEL

GUI-only remainder: dynpro screens + GUI statuses (SE51/SE41). The code compiles headless; a human paints the trivial screen. As tested, the lab is 100% headless-buildable — no Eclipse, no VSP, no GUI required for authoring.
